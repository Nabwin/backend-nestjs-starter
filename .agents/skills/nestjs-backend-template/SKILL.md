# Backend Template — Coding Rules

## Imports

### No `.js` Extensions

NEVER use `.js` extensions in import paths. The project uses `src/*` path aliases and Bun runtime — no ESM file extensions needed.

```ts
// WRONG
import { PrismaService } from "./prisma.service.js";
import { PrismaClient } from "./generated/client.js";

// CORRECT
import { PrismaService } from "./prisma.service";
import { PrismaClient } from "./generated/client";
import { PrismaService } from "src/db/prisma.service";
```

The only `.js` extensions you'll see are in `src/db/generated/` — those are auto-generated by Prisma and gitignored. Do NOT modify them or copy that pattern.

### `src/*` Path Aliases

Use `src/*` path aliases (configured in `tsconfig.json`) for cross-module imports:

```ts
import { PrismaService } from "src/db/prisma.service";
import { OAUTH_METHOD } from "src/oauth/types/enums";
```

No relative `../../../` chains across modules.

## Enums — No Magic Strings

### Use Prisma Enums, Not Raw Strings

When comparing against Prisma enum values, import the enum from `src/db/generated/enums` and use the enum member — never a raw string.

```ts
// WRONG — magic string
if (user.status !== "ACTIVE") { ... }

// WRONG — magic string via PrismaService
if (user.status !== "ACTIVE") { ... }

// CORRECT
import { UserStatus } from "src/db/generated/enums";
if (user.status !== UserStatus.ACTIVE) { ... }
```

Available Prisma enums:
- `UserStatus.ACTIVE | UserStatus.SUSPENDED | UserStatus.BANNED`
- `OAuthProvider.GOOGLE | OAuthProvider.FACEBOOK | OAuthProvider.APPLE | OAuthProvider.LINKEDIN`

### Project Enums

Use the project enum objects for type-safe comparisons:

```ts
import { OAUTH_METHOD } from "src/oauth/types/enums";
import { DEVICE_TYPE } from "src/oauth/types/enums";

// WRONG — magic string
deviceType: "web"
provider: "GOOGLE"

// CORRECT
deviceType: DEVICE_TYPE.WEB
provider: OAUTH_METHOD.GOOGLE
```

No raw strings like `"admin"`, `"user"`, `"web"`, `"mobile"`, `"GOOGLE"`, `"active"`, etc. in comparisons or assignments.

## Prisma Usage

### PrismaService in NestJS (DI)

Inside NestJS services/controllers, inject `PrismaService` — never create a raw `PrismaClient`.

```ts
// CORRECT — NestJS service
@Injectable()
export class MyService {
  constructor(private readonly prisma: PrismaService) {}
  
  async findUser(email: string) {
    return this.prisma.user.findUnique({ where: { email } });
  }
}
```

### PrismaClient in Scripts (seeders, standalone scripts)

Only in seeders/scripts outside the NestJS DI container, create PrismaClient directly with the adapter:

```ts
// CORRECT — seeder/script
import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "../src/db/generated/client";

const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL });
const prisma = new PrismaClient({ adapter });
```

### `PrismaModule` is Global

`PrismaModule` is `@Global()` — only imported in `app.module.ts`. Do NOT add it to sub-module imports.

### No Raw SQL

All database access goes through Prisma's type-safe query builder. No `$queryRaw`, `$executeRaw`, or template strings with SQL.

## Migration Workflow

NEVER run `prisma migrate dev` directly — it auto-applies migrations, which is dangerous.

### Step 1: Create migration (does NOT apply)

```sh
bunx prisma migrate dev --create-only --name descriptive_name
```

Review the generated SQL in `prisma/migrations/<timestamp>_descriptive_name/migration.sql`.

### Step 2: Apply migration

```sh
bunx prisma migrate deploy
```

### Full flow:

```sh
# Make schema changes in prisma/schema.prisma first
bunx prisma migrate dev --create-only --name create_tables
# Review migration.sql
bunx prisma migrate deploy
```

This applies only unapplied migrations — safe for CI/CD.

## Database Seeding

### Seed Structure

Seeders live in `prisma/seeders/` with numeric prefixes for ordering:

```
prisma/
  seed.ts              → entry point: scans seeders/, runs each `up()` function
  seeders/
    000000_create_roles.ts   → creates roles (skips if already seeded)
    000001_create_users.ts   → creates users linked to roles
```

### Seeder Pattern

Each seeder exports an `up` function:

```ts
import type { PrismaClient } from "../../src/db/generated/client";

export const up = async (prisma: PrismaClient) => {
  // idempotent — check if data exists first
  const count = await prisma.role.count();
  if (count) return;

  await prisma.role.createMany({
    data: [
      { name: "Admin", slug: "admin", permissions: ["*"] },
      { name: "User", slug: "user", permissions: [] },
    ],
  });
};
```

### Running Seeds

```sh
bun db:seed
# or directly:
bun run prisma/seed.ts
```

Seeds are idempotent (check before creating) — safe to run multiple times.

## Security Rules

### Password Handling
- Always use `bcrypt.compare()` for password verification (timing-safe)
- Always use `bcrypt.hash()` with at least 12 salt rounds
- Never store plain-text passwords

### Token Handling
- Refresh tokens: SHA-256 hash before storage
- JTI blacklisting on logout
- Rotation detection (reuse = revoke all sessions)

### Error Messages
- Never expose internal error details to clients
- Log the actual error server-side, return generic messages
- OAuth callback errors: use `this.logger.error()` and return generic user-facing error

### Rate Limiting
- Login attempts tracked per email + per IP via Redis
- Redis failures should NOT block legitimate traffic — allow through on Redis error

### OAuth State Parameter
- Do NOT trust client-supplied state values for sensitive data without integrity verification

## Testing

### Test Framework

Use **supertest + Jest** for e2e tests. Tests go in `src/**/*.spec.ts`.

### Test Setup

```ts
import { Test, TestingModule } from "@nestjs/testing";
import { INestApplication } from "@nestjs/common";
import * as request from "supertest";

describe("Auth (e2e)", () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleFixture: TestingModule = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();
    app = moduleFixture.createNestApplication();
    await app.init();
  });

  afterAll(async () => {
    await app.close();
  });
});
```

## Quick Reference

| Do | Don't |
|---|---|
| `import { UserStatus } from "src/db/generated/enums"` | `user.status !== "ACTIVE"` |
| `import { PrismaService } from "src/db/prisma.service"` | Create raw `PrismaClient` in NestJS services |
| `prisma migrate dev --create-only` then `migrate deploy` | `prisma migrate dev` directly |
| `import type { PrismaClient } from "../../src/db/generated/client"` in seeders | `.js` extensions in imports |
| `bcrypt.compare(plain, hash)` | `===` for password comparison |
| `this.logger.error(msg, stack)` + generic user error | Raw error messages to clients |
| Enum keys like `OAUTH_METHOD.GOOGLE`, `DEVICE_TYPE.WEB` | Magic strings like `"GOOGLE"`, `"web"` |
